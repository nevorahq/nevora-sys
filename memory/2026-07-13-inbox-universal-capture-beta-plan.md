# Design: Universal Capture Inbox for Beta

Generated by office-hours on 2026-07-13  
Status: DRAFT  
Mode: Startup / pre-product beta

## Problem Statement

The current Action Center empty state offers document upload and Inbox capture.
Those controls are technically links to module-owned flows, but they make the
Action Center look like another creation surface. The ownership model is unclear:
Action Center, Inbox, Documents, and the First Action Wizard all appear able to
start the same workflow.

The beta needs one understandable loop:

`capture in Inbox -> process in the owning module -> review -> confirm -> execute`

## Product Ownership Contract

1. **Inbox owns capture and capture history.** It accepts text, a camera photo, or
   document files and shows their processing state.
2. **Documents owns every uploaded file.** Inbox never introduces a second file
   store, attachment table, extraction engine, or upload policy.
3. **Action Center owns attention, not creation.** It lists work and review
   signals. A planner signal opens the exact Inbox review instead of creating an
   unrelated generic task draft.
4. **Tasks, Money, Subscriptions, and Relations own accepted records.** Existing
   services remain the only paths that create those records.
5. **AI proposes; the user confirms.** A photo or document may create a draft or
   suggestion, but it never posts a money transaction.

## Beta Scope

### Included

- Remove creation CTAs and the large `soft-inset` empty state from Action Center.
- Remove or relocate the First Action Wizard from Action Center so the page has no
  document/task/subscription creation controls.
- Add `Text`, `Photo`, and `Document` modes to the Inbox capture composer.
- Reuse Documents file validation, camera input, upload, storage, extraction,
  billing limits, audit events, and rollback behavior.
- Create one `planner_entry` linked by `source_document_id` for each successful
  Inbox photo/document capture.
- Show capture states: uploading, processing, ready for review, failed, accepted,
  and archived.
- Deep-link Action Center planner signals to the exact Inbox suggestion.
- Preserve the existing Inbox accept/edit/reject and exactly-once confirmation
  services for beta.
- Add idempotency for retried Inbox uploads and document-to-planner linking.
- Support mobile camera capture through the existing HTML capture input, with the
  platform file/photo picker as the compatibility behavior.

### Explicitly excluded

- A custom `getUserMedia()` camera UI.
- Voice, URL, or email forwarding capture.
- A second Inbox-specific file store.
- Private file visibility separate from Documents.
- Automatic understanding of DOCX, HEIC, or HEIF. These files may be stored, but
  automatic extraction remains limited by the existing extractor.
- Autonomous task, subscription, or posted transaction creation.
- Removing all module-specific review systems in favor of one new review engine.

## User Experience

### Inbox composer

Use a compact segmented mode control:

- **Text:** the current textarea and Capture command.
- **Photo:** camera command, one image preview, optional context note.
- **Document:** file command, existing multi-file preview and limits, optional
  context note.

Photo/document capture should not require a manually written title. Generate a
safe title from the first filename or `Photo capture <date>`. Show the message
`Saved to Documents for <organization>` before submission because Documents are
organization-visible even though planner entries are private to their owner.

After submission, keep the user in Inbox and add a capture card immediately:

- `Uploading` while bytes are being sent.
- `Processing` after the Document exists and extraction is queued.
- `Review ready` when a suggestion or draft exists.
- `Needs manual review` for unsupported or low-confidence extraction.
- `Failed` with Retry when upload or linking did not complete.

### Action Center

- Empty state: a compact neutral message, `No actions require your attention`.
- No Upload document, Capture note, Create task, or Add subscription CTAs.
- A planner-backed `ai_suggestion` shows `Open review`, linking to
  `/dashboard/inbox?tab=review&suggestion=<id>`.
- Do not offer `Create task draft` for an item whose primary entity is a
  `planner_entry` or `planner_suggestion`.
- Generic non-planner AI signals may retain their existing task-draft action.

## Target Flow

### Text

1. Inbox creates a text `planner_entry`.
2. Existing intent detection creates a `planner_suggestion`.
3. Existing Action Center materialization creates one attention signal.
4. The user opens the exact review and accepts, edits, or rejects it.
5. Existing module services create the accepted entity exactly once.

### Photo or document

1. Client creates a stable capture request UUID and submits files plus optional
   context to an Inbox upload endpoint.
2. The endpoint checks both planner and document permissions, billing access,
   storage quota, file count, type, and size.
3. A shared Documents service creates the Document and attachments. The existing
   `/api/documents/upload` route must use the same service.
4. Inbox creates or reuses one sourced `planner_entry` with `source=document`,
   `source_document_id=<document id>`, and the correct entry type (`photo` or
   `document`).
5. Readable PDF/image captures enter the existing extraction pipeline. Unsupported
   files remain valid Documents and produce a manual-review state.
6. The resulting planner or financial review signal is materialized in Action
   Center without creating a second business record.
7. Confirmation routes through the existing owning service. No upload or
   extraction path posts money.

## Data and Idempotency

Use the next free migration number at implementation time (currently `105`).

Recommended additions:

- Nullable `documents.inbox_capture_id uuid`.
- Unique index on `(organization_id, created_by, inbox_capture_id)` when the token
  is present, so network retries reuse the same Document.
- Unique partial index on
  `(organization_id, owner_user_id, source_document_id)` for planner entries with
  a document source, so one Document cannot seed duplicate Inbox entries.
- A server-side helper that resolves an existing Document and planner entry before
  attempting another insert.

Do not add attachment blobs or extraction JSON to `planner_entries`; it stores only
the source pointer and human-readable capture context.

## Development Phases

### Phase 0: Contract tests and baseline

- Encode the ownership contract in tests before changing UI.
- Prove the current text capture and planner accept paths still pass.
- Add regression coverage showing a planner Action Center item must not execute
  the generic `create_task_draft` path.

Exit: baseline is green and the current incorrect planner quick action is covered
by a failing regression test.

### Phase 1: Clean Action Center ownership

- Replace `ActionEmptyState` with a compact, action-free message.
- Remove `showFirstActions` and creation CTAs from the Action Center feed.
- Move First Action Wizard entry points to Inbox and module-owned empty states, or
  retire the wizard UI while preserving activation telemetry.
- Stop advancing onboarding through Action Center render. Record activation after
  successful source-module creation instead.
- Change planner-backed Action Center actions to `Open review` with an exact Inbox
  deep link.

Exit: there is no creation control anywhere on `/dashboard`; planner signals open
their real source review.

### Phase 2: Extract a reusable Documents upload service

- Move the create/upload/rollback/event/extraction-enqueue logic out of
  `/api/documents/upload` into one Documents domain service.
- Keep the current route as a thin adapter to protect the Documents screen.
- Return a typed result containing document id, attachment ids, whether extraction
  was queued, and a recoverable warning when post-upload Inbox linking fails.
- Preserve all existing billing, permission, redaction, audit, and cleanup rules.

Exit: the existing Documents upload behaves identically and its route tests pass;
no upload/storage loop is copied into Planner.

### Phase 3: Build the multimodal Inbox composer

- Replace `CaptureInput` with an `InboxCaptureComposer` using Text/Photo/Document
  modes.
- Reuse `DocumentFileUpload`, `useDocumentFiles`, and document validation constants.
- Add optional context, previews, remove/retry behavior, loading state, and
  bilingual copy.
- Keep dimensions stable on mobile and desktop; camera and file buttons use
  Lucide icons and accessible labels.

Exit: text behavior is unchanged; supported photos/files can be selected,
validated, previewed, and submitted from Inbox.

### Phase 4: Add Inbox document capture orchestration

- Add a dedicated binary upload endpoint for Inbox rather than teaching the text
  Server Action to handle files.
- Generate and send a stable capture request id from the client.
- Call the shared Documents service, then create/reuse the sourced planner entry.
- Revalidate Inbox, Documents, Action Center, and relevant review surfaces.
- Make partial outcomes explicit: a successfully stored Document is never deleted
  solely because planner linking failed; return a warning and allow repair/retry.
- Add a reconciler for Documents carrying an Inbox capture id but missing their
  planner entry.

Exit: one request/retry produces exactly one Document and one linked planner entry.

### Phase 5: Processing and review presentation

- Hydrate Inbox cards with linked Document title, type, attachment count, and
  extraction status.
- For supported financial files, reuse the existing document extraction and
  financial review result; do not create a parallel financial fact.
- For non-financial or unreadable captures, create a deterministic manual-review
  suggestion such as `Review <document>` rather than pretending content was
  understood.
- Make `?tab=review&suggestion=<id>` select the Review tab and focus the exact
  suggestion.
- Ensure accepting/rejecting resolves the linked Action Center item.

Exit: every successful capture reaches a visible ready/manual-review state, and
every Action Center planner signal opens the correct review.

### Phase 6: Beta hardening and QA

- Unit tests for capture mode selection, file validation, sourced entry creation,
  idempotency, deep-link routing, permissions, and money safety.
- Route/service tests for full success, quota denial, unsupported extraction,
  storage failure rollback, planner-link failure, and network retry.
- Integration tests for text, camera image, PDF invoice, generic PDF, and
  unsupported-but-stored DOCX/HEIC.
- Browser QA at mobile and desktop widths for no overlap, preview stability,
  keyboard navigation, camera fallback, and clear organization-visibility copy.
- Run typecheck, lint, unit tests, build, database verification, and the release
  smoke path.

Exit: all beta success criteria below are demonstrated with automated tests or a
recorded manual smoke result.

## Test Matrix

| Scenario | Expected result |
|---|---|
| Text capture | Existing suggestion appears once and remains editable/rejectable |
| Mobile photo | One Document and one linked photo entry; picker/camera fallback works |
| PDF invoice | Extraction queues; review appears; no posted transaction |
| Generic PDF | Document is stored; honest review task/manual state appears |
| DOCX or HEIC | File is stored; automatic extraction is marked unavailable |
| Retry same request | No duplicate Document, planner entry, suggestion, or action item |
| Storage failure | Uploaded partial objects and Document row are rolled back |
| Planner link failure | Document remains safe; warning and repair path are available |
| No permission/quota | Request fails before storage; no partial records |
| Empty Action Center | Neutral message only; no creation CTA or wizard |
| Planner signal in Action Center | Opens exact Inbox review; never creates generic follow-up task |
| Accept twice | Exactly one task/reminder/link is created |

## Success Criteria

- A first-time user can identify Inbox as the only quick capture surface without
  explanation.
- `/dashboard` contains zero creation CTAs in both empty and populated states.
- Text, photo, and document capture all begin from Inbox.
- Every uploaded file appears exactly once in Documents and every Inbox-origin
  Document has exactly one linked planner entry.
- A planner review signal always returns to the original capture and never creates
  an unrelated follow-up task.
- Unsupported extraction is honest and recoverable; the capture is never silently
  lost.
- Upload, extraction, and confirmation cannot create a posted money transaction.
- Existing Documents and text Inbox workflows do not regress.

## Dependencies and Risks

- **Visibility:** Documents are organization-visible while planner captures are
  private. Beta copy must disclose this before upload.
- **Extraction coverage:** the current router automatically reads PDF, PNG, JPG,
  JPEG, and WEBP; DOCX/HEIC/HEIF need manual review.
- **Review duplication:** keep the existing Inbox Review tab for beta, but Action
  Center must deep-link instead of inventing a second planner action. Re-evaluate
  whether the tab remains after observing beta users.
- **Onboarding telemetry:** removing the wizard from Action Center must not remove
  activation measurement. Move tracking to successful domain actions.
- **Render-time generation:** Action Center still syncs action items during render.
  This plan avoids increasing that dependency; moving generation to events/cron is
  a separate production-hardening task.
- **AI availability:** capture remains durable when AI fails. The UI must show a
  retry/manual-review state rather than block upload.

## Open Questions

1. Whether beta users expect Inbox-uploaded Documents to be shared with the whole
   organization. If not, private Documents are a separate data-model project and
   must block this design before implementation.
2. Whether generic non-financial documents need semantic AI understanding in the
   first beta, or whether `Review document` is an acceptable honest fallback.
3. Whether the current Inbox Review tab should remain after beta observation or
   Action Center should eventually become the only review queue.

## Recommended Delivery Order

Ship as two internal increments, but release them together:

1. **Ownership correction:** clean Action Center, fix planner signal routing, move
   onboarding writes out of Action Center render.
2. **Universal capture:** shared Documents service, multimodal Inbox composer,
   linked planner entries, idempotency, processing states, and QA.

Estimated implementation effort: **6-10 focused engineering days**, depending on
how much of the upload route can be extracted without changing its behavior and
whether generic-document AI understanding is required for beta.

## The Assignment

Before implementation, run one paper test with a person who has not seen Nevora:
show only the sidebar and ask, `Where would you add a photographed invoice?` The
design passes if they choose Inbox immediately, then understand that the saved
file lives in Documents and the resulting decision appears as a review signal.

## What I Noticed

The strongest product signal in this review was the insistence that creation
belongs to Documents, Subscriptions, and Inbox rather than Action Center. That is
not cosmetic feedback; it defines the operating model. The code already contains
most of the right primitives, but the current first-action UI makes their ownership
look less coherent than the underlying architecture actually is.
